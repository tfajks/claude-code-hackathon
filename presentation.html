Coordinator + Specialist Architecture

Design Decision #1

Why two agents, not one?

Tool count ceiling

A single agent needs all 5 tools. Reliability drops past ~5 tools. The classifier runs with 2 tools only — lookup_kb and get_user_history.

Context isolation

Classification can't be biased by knowing which actions are available. Specialist receives only ticket text + user_id. Nothing from coordinator context leaks in.

Cost: ~60% savings

Haiku for classification (structured extraction).
Sonnet for coordination (multi-step reasoning).
~$0.80–$1.50/day at 200 tickets/day.

Pattern: Task tool

Specialist invoked as a function call — same pattern as Claude Agent SDK's Task tool. Context passed explicitly. No shared state between agents.

Design Decision #2 — ADR-001

Hooks vs. Prompts for safety

Guardrail Mechanism Reason
Frozen account blocking HOOK (Python) Deterministic — must block regardless of model confidence
PII in ticket content HOOK (regex) Regex is 100% reliable for known PII patterns; LLM has false negatives
High-risk auto-resolve HOOK (keyword) Privilege keywords in execute_resolution never acceptable to auto-approve
Urgency inflation PROMPT Probabilistic judgment — "CRITICAL EMERGENCY" may or may not be real
Prompt injection resistance PROMPT Hook can't detect all injection forms; model instruction + training handles it
Confidence threshold CODE (Python) Deterministic once confidence value returned; < 0.70 always escalates

Rule: Hooks handle conditions where the answer is always the same regardless of context. Prompts handle conditions requiring judgment.

Agent Logic

Escalation rules — deterministic, any one triggers

ALWAYS ESCALATE IF
  • 🔴 Category is security_breach, malware, ransomware, account_compromise…
  • 🟠 Confidence < 0.70 (agent uncertain)
  • 🟡 Affected users > 50 (potential outage)
  • 🟡 Estimated impact = "high"
  • 🔴 security_flag = true
AUTO-RESOLVE ONLY IF ALL TRUE
  • ✅ auto_resolvable = true
  • ✅ confidence ≥ 0.85
  • ✅ action in: sspr_link | vpn_client | approved_software
  • ✅ security_flag = false
  • ✅ estimated_impact = "low"
NEVER AUTOMATE
  • 🚫 Privilege escalation (admin, sudo, elevated perms)
  • 🚫 Actions on frozen/terminated accounts
  • 🚫 Security incident investigation
  • 🚫 Data recovery from backups
  • 🚫 Executive account changes
  • 🚫 Multi-system access requests
  • 🚫 Unverifiable claimed authority

Reliability Pattern

Validation-retry loop

# coordinator.py — MAX_RETRIES = 3
for attempt in range(MAX_RETRIES):
    raw = _run_coordinator_loop(
        ticket_text, user_id,
        classification,
        forced_escalate,
        client,
        last_error    # ← fed back on retry
    )

    try:
        result = TriageResult.model_validate_json(raw)
        return result, reasoning_log

    except ValidationError as e:
        last_error = str(e)
        reasoning_log["retry_count"] += 1
        reasoning_log["retry_errors"].append(
            {"attempt": attempt+1,
             "error": last_error}
        )

raise RuntimeError("max retries exceeded")

Why this matters

On retry, the exact Pydantic validation error is fed back to the model as a user message. The model knows precisely what to fix — not just "try again."

Full reasoning log

Every triage run writes a JSON log with: classification, forced_escalate decision, retry count + errors, final result. Every decision is replayable from the log alone.

stop_reason handling

Agent loop checks stop_reason on every turn: end_turn → extract JSON, tool_use → dispatch + continue, other → break (not an iteration cap).

Tool Design

5 tools, structured error responses

CLASSIFIER TOOLS (2)
lookup_kb
Knowledge base search. No live system access.
get_user_history
Last 5 tickets. No PII, no cross-user data.
COORDINATOR TOOLS (5, incl. above)
create_ticket · assign_to_team · execute_resolution 
# Structured error — agent can retry, not parse a string
{
  "isError": true,
  "reason_code": "INVALID_TEAM",
  "guidance": "Unknown team_id 'exec2'.
    Valid values: desktop, network,
    security, iam, email, data, exec, tier2"
}

# Hook decision — same structure as tool error
{
  "isError": true,
  "reason_code": "FROZEN_ACCOUNT",
  "guidance": "Account 'frozen_usr_001' is
    suspended or terminated. No actions
    can be taken. Contact HR and Security."
}

Consistent error shape: isError + reason_code + guidance. Model doesn't parse strings — it reads structured recovery guidance.
